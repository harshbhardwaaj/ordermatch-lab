# Copy Principles

**Date**: July 2, 2026
**Purpose**: Keep product, candidate, and CTA writing direct, specific, and safe.

## Voice

OrderMatch Lab should sound like a sharp builder explaining what he studied and built. The writing should be:

- direct
- confident
- specific
- plain-spoken
- product-minded
- honest about what is simulated
- serious without sounding stiff

The product should sell Harsh through evidence, not inflated claims.

## Core Sentence

Use this as the north star:

> I studied Comena's order-entry workflow and built OrderMatch Lab to make the hard parts visible: extraction, SKU matching, confidence, exception review, ERP readiness, onboarding, traceability, and evals.

## Comena Framing

Use Comena explicitly in:

- opening section
- final CTA
- source/research context where relevant

Use reusable product language in:

- core prototype
- order review
- evals
- onboarding/setup

This keeps the experience built for Comena first and adaptable later.

## Candidate Tone

Candidate proof should be selective and tied to Comena needs.

Good pattern:

> Built and deployed a full-stack finance tool with a Python backend and Next.js frontend. Relevant here because Comena needs interns who can connect UI, business logic, and shipped deployment.

Weak pattern:

> I am passionate about AI and eager to learn.

Candidate content should favor:

- ALEVOR AI classification and eval work
- AI Investment Analyst full-stack deployment
- CV-JD Fit Scorer document and structured-output workflow
- relevant TUM coursework without grades

## No-Overclaim Rules

Do not imply:

- access to Comena internal systems
- access to Comena customer data
- access to Comena product code
- benchmark results against Comena
- real ERP integration in early versions
- live extraction/matching before backend work exists
- production automation where behavior is simulated

Use safer phrases:

- research-informed
- grounded synthetic data
- sample order
- prototype interaction
- simulated review decision
- future backend boundary
- designed for later API-backed functionality

## Writing Patterns

Use:

- "This blocks ERP readiness because..."
- "The system preserved the original text and normalized..."
- "This match is review-needed because..."
- "No match found. Route to human review or create catalog item."
- "Sample eval run complete."
- "Calendar link unavailable. Use email instead."

Avoid:

- "Something went wrong."
- "AI magic."
- "Autonomous agent solved it."
- "Revolutionary."
- "Seamless" unless it is a source quote.
- "Next-generation."
- "Transformative."
- "End-to-end" unless describing a specific built flow.

## Banned Words And Phrases

Scope note: these rules apply to app copy, CTA copy, candidate proof, README/send copy, and other reviewer-facing text. They are guidance, not a hard lint rule, for internal planning, research, and design notes.

Do not use these in product/candidate copy unless discussing the banned list itself:

- synergy
- leverage as a verb
- passionate
- thrilled
- excited to apply
- detail-oriented
- team player
- hardworking
- results-driven
- dynamic
- innovative
- spearheaded
- utilized
- impactful
- cutting-edge
- additionally
- showcase
- testament
- underscore
- vibrant
- pivotal
- crucial
- delve
- fostering
- align with
- landscape as an abstract noun
- tapestry
- intricate
- garner
- highlight as a verb
- interplay
- enduring

- emphasizing

Avoid em dashes and en dashes across the project. Use commas or periods.

## Simulated Behavior Labels

Use compact labels, not apologetic disclaimers:

- Sample data
- Prototype action
- Simulated eval
- Demo catalog
- API-ready shape

Example:

> Prototype action: accepting this match updates the sample order state.

Avoid:

> This is just mocked and not real.

## Error Copy Formula

A useful error message should say:

1. what happened
2. why it matters
3. what to do next

Examples:

- "The document preview did not load. Extracted fields are still available, so you can continue review or retry the preview."
- "This order cannot be marked ERP-ready yet. Two line items still have blocking SKU exceptions."
- "The sample eval failed to run. Keep the last metrics visible and retry the benchmark."

## Empty State Formula

An empty state should say:

1. why the area is empty
2. what will appear here
3. the next useful action

Examples:

- "No orders are waiting for review. Import sample orders to inspect the workflow."
- "No eval runs yet. Run the sample benchmark to establish a baseline."
- "No unresolved exceptions. This order is ready for final validation."

## CTA Direction

Final CTA should be direct:

> If this direction is useful, I would like to intern with Comena and build the real version with your team.

Support with:

- quick call
- email fallback
- GitHub
- LinkedIn
- relevant projects
- resume

Do not make the ending timid or vague.
